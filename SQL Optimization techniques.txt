Here’s a concise but comprehensive list of SQL query optimization techniques that can help improve performance across most relational databases (MySQL, PostgreSQL, SQL Server, Oracle, etc.):

SQL Query Optimization Techniques
1. Index Optimization

Create indexes on columns used in WHERE, JOIN, ORDER BY, and GROUP BY.
Use covering indexes to avoid extra table lookups.
Avoid over-indexing (too many indexes slow down INSERT/UPDATE/DELETE).
Use composite indexes for multi-column filtering.


2. Query Structure Improvements

Select only required columns (SELECT specific_columns) instead of SELECT *.
Use EXISTS instead of IN for subqueries when possible.
Replace OR with UNION or IN if it improves index usage.
Avoid unnecessary nested subqueries; use JOINs instead.
Use CASE instead of multiple IF or OR conditions when possible.


3. JOIN Optimization

Ensure JOIN columns are indexed.
Use the correct JOIN type (INNER JOIN vs LEFT JOIN) to avoid extra rows.
Filter rows before joining (apply WHERE early).


4. Filtering & Aggregation

Apply filters early in the query to reduce dataset size.
Use HAVING only for aggregated conditions; use WHERE for non-aggregated filters.
Avoid functions on indexed columns in WHERE (e.g., WHERE YEAR(date) = 2024 → use range filtering instead).


5. Execution Plan Analysis

Use EXPLAIN or EXPLAIN ANALYZE to check query execution plans.
Identify full table scans and replace them with indexed lookups.
Check for unnecessary sorting or temporary table creation.


6. Data & Schema Optimization

Normalize data to reduce redundancy, but denormalize for heavy read queries if needed.
Partition large tables for faster access.
Use appropriate data types (smaller types = less memory & faster scans).


7. Caching & Materialization

Cache frequent query results in memory or materialized views.
Use query result caching in the application layer if data changes infrequently.


8. Transaction & Locking

Keep transactions short to avoid locking delays.
Use batch inserts/updates instead of row-by-row operations.

Views vs CTEs vs Temporary Tables — Performance Impact



Feature
When It Helps Performance
When It Hurts Performance




Views
- Good for code reuse and simplifying complex joins. <br> - Indexed Views (materialized views) can speed up queries.
- Normal (non-indexed) views are just query wrappers — they don’t store data, so performance is same as running the query directly.


CTEs (Common Table Expressions)
- Good for readability and breaking down complex queries. <br> - Useful for recursive queries.
- CTEs are recomputed every time they are referenced (in most DBs), so they don’t inherently improve speed.


Temporary Tables
- Can store intermediate results and reuse them multiple times in a session. <br> - Useful when working with large datasets that need multiple passes.
- Writing to temp tables has overhead. <br> - Overuse can cause tempdb contention in SQL Server or similar issues in other DBs.




Other Formats for Query Optimization
1. Inline Subqueries
SqlSELECT e.EmployeeID, e.Name
FROM Employees e
WHERE e.DepartmentID IN (
    SELECT DepartmentID
    FROM Departments
    WHERE Location = 'New York'
);

✅ Good for simple filters.
❌ Can be slow if subquery is large and not indexed.

2. JOIN with Filtering
SqlSELECT e.EmployeeID, e.Name
FROM Employees e
JOIN Departments d ON e.DepartmentID = d.DepartmentID
WHERE d.Location = 'New York';

✅ Often faster than IN because it can use indexes on join columns.

3. Materialized Views (Indexed Views)
Sql-- Create a materialized view (syntax varies by DB)
CREATE MATERIALIZED VIEW emp_ny AS
SELECT e.EmployeeID, e.Name
FROM Employees e
JOIN Departments d ON e.DepartmentID = d.DepartmentID
WHERE d.Location = 'New York';

✅ Stores results physically → fast retrieval.
❌ Needs refresh when data changes.

4. Using Temporary Tables
Sql-- Step 1: Store filtered departments
CREATE TEMP TABLE ny_departments AS
SELECT DepartmentID
FROM Departments
WHERE Location = 'New York';

-- Step 2: Join with employees
SELECT e.EmployeeID, e.Name
FROM Employees e
JOIN ny_departments nd ON e.DepartmentID = nd.DepartmentID;

✅ Great when the filtered set is reused multiple times in a session.

5. Using CTE
SqlWITH ny_departments AS (
    SELECT DepartmentID
    FROM Departments
    WHERE Location = 'New York'
)
SELECT e.EmployeeID, e.Name
FROM Employees e
JOIN ny_departments nd ON e.DepartmentID = nd.DepartmentID;

✅ Improves readability.
❌ No inherent performance gain unless DB engine optimizes it internally.

Quick Recommendation

For speed: Use indexed tables/materialized views or temp tables when reusing data multiple times.
For readability: Use CTEs.
For abstraction: Use views (indexed if performance is critical).
